# Experiment Report Runner

## Goal

Use an already implemented repository to:
1. identify the experiments the project requires,
2. inspect the existing result files in `outputs/`,
3. collect the metrics/artifacts already produced by the user,
4. and write the final project report into `docs/report/report.tex`.

This skill assumes the system already exists and the experiment outputs were already generated by the user.
This skill is **not** for designing the architecture from scratch.
This skill is **not** for running experiments from scratch unless the user explicitly overrides this rule.

## When to use

Activate this skill when the user:
- already has an implemented ML/DL project repository,
- already ran experiments manually or earlier,
- wants the coding agent to inspect `outputs/` instead of executing training/evaluation,
- wants all reportable experiments described in the README to be checked against available outputs,
- wants the final report to be written from actual obtained results,
- already has a report outline/template in `references`,
- wants the LaTeX writing to follow another skill such as `latex-writing-skill`.

## Required inputs

The repository should contain or provide:
- a `README.md` describing the expected experiments and workflow,
- the implemented codebase,
- an `outputs/` directory containing experiment artifacts generated by the user,
- optionally configs / scripts / entrypoints for understanding result provenance,
- a project plan or implementation plan if relevant,
- a report outline/template in `references` or an equivalent attached resource,
- optionally an existing `docs/report/report.tex` to update.

## Source-of-truth priority

Use sources in this order:
1. explicit user instructions,
2. the project plan / implementation plan,
3. the README experiment instructions,
4. the report outline/template in `references`,
5. repository code/config reality,
6. the actual files inside `outputs/`.

For **what should be reported**, use README/project docs.
For **what actually happened**, use only the actual artifacts found in `outputs/` and directly derived summaries from those artifacts.

If the README says an experiment should exist but there is no corresponding result in `outputs/`, mark it as missing/incomplete rather than inventing or rerunning it.

## Core rules

1. **Report what exists, not what could be rerun.**
   Do not run experiments by default. Use only the artifacts already present in `outputs/` unless the user explicitly asks for execution.

2. **Treat README as the expected experiment checklist.**
   The README defines what experiments should be represented in the final report.

3. **Treat `outputs/` as the evidence base.**
   Every result, metric, figure, and claim in the report must come from actual files in `outputs/` or from direct summaries computed from them.

4. **Do not invent missing results.**
   If an expected experiment has no matching outputs, explicitly mark it as missing, incomplete, or unavailable.

5. **Prefer minimal repository intervention.**
   Do not redesign the repository and do not add new execution logic just to fill report gaps.

6. **Use `latex-writing-skill` for LaTeX quality.**
   When writing or revising `docs/report/report.tex`, follow the conventions and style rules from `latex-writing-skill`.

## Execution workflow

### 1. Inspect the repository and documentation
Read:
- `README.md`
- project plan / implementation plan if present
- report template / report outline from `references`
- existing `docs/report/report.tex` if it already exists
- any minimal code/config pieces needed to understand what the artifacts in `outputs/` mean

Extract:
- the expected experiment list
- expected workflow stages
- expected outputs and output naming/location conventions
- which report sections correspond to which experiments
- what files/metrics/plots should be searched for in `outputs/`

### 2. Build an expected experiment checklist
Create a concrete checklist of all report-relevant experiments.
For each experiment, identify:
- name
- purpose
- expected evidence in `outputs/`
- likely file types or directories
- report section/subsection it should inform

Do not add extra experiments unless they are necessary to fulfill the stated plan.

### 3. Inspect `outputs/`
Traverse `outputs/` and map actual artifacts to the expected experiment checklist.
For each candidate artifact, identify when possible:
- experiment/run name
- config or variant identity
- metric files
- plots/figures
- tables/summaries
- logs useful for provenance
- final saved outputs relevant to the report

Be conservative in matching files to experiments.
If mapping is ambiguous, state the ambiguity instead of guessing.

### 4. Build a results inventory
For each expected experiment, classify it as:
- evidence found and reportable,
- partially evidenced,
- missing,
- ambiguous.

For reportable experiments, capture:
- artifact paths
- key metrics
- figures/tables available
- brief grounded interpretation
- any caveats about completeness or comparability

### 5. Derive report material from existing artifacts
From the actual files in `outputs/`, gather the material needed for the report:
- final metrics
- comparison tables
- plots/figures
- qualitative observations if relevant
- reproducibility details that can be grounded in config/log files
- missing-result notes where expected outputs are absent

If multiple runs/seeds exist, summarize them consistently with the repository conventions.
If aggregation is needed, compute it only from the existing output artifacts.

### 6. Write the report incrementally
Use the report structure/template from `references` as the scaffold.
For each experiment with evidence in `outputs/`:
- identify the corresponding report section,
- add the relevant result summary,
- describe the setup briefly based on repository/docs plus available artifacts,
- include grounded interpretation,
- reference produced artifacts/tables/figures as appropriate.

For each expected experiment without sufficient evidence:
- add a concise note that the experiment was expected,
- state that no adequate output artifacts were found,
- do not fabricate quantitative or qualitative conclusions.

### 7. Finalize `docs/report/report.tex`
Produce or update `docs/report/report.tex` so that it becomes the final project report.
The report should:
- follow the provided structure,
- be based on actual discovered outputs,
- clearly distinguish available vs missing evidence,
- remain technically precise and concise,
- use proper LaTeX style guided by `latex-writing-skill`.

## Report-writing rules

When writing `docs/report/report.tex`:
- use the provided report outline/template as the required structure,
- do not invent sections that contradict the template,
- prefer concise technical writing over inflated prose,
- state exact experiment settings only when they can be grounded in README/config/log artifacts,
- keep interpretation grounded in found results,
- explicitly mention limitations, missing experiments, or ambiguous artifact mapping when relevant,
- ensure tables/figures/results referenced in text actually exist or are explicitly marked unavailable.

## Allowed minimal fixes

Minimal repository changes are allowed only if they are necessary to complete the report-writing workflow, for example:
- creating a missing `docs/report/` directory,
- creating an initial `docs/report/report.tex` from the provided template,
- fixing a trivial broken relative path in report references,
- organizing references to already existing artifacts,
- adding a tiny helper script only if strictly necessary to read/aggregate existing output files.

Do **not**:
- rerun experiments by default,
- change training code just to generate missing outputs,
- modify result artifacts,
- perform broader refactors or speculative cleanup.

## What to avoid

Avoid the following:
- running experiments unless the user explicitly asks for it,
- inventing experiments that are not required,
- claiming results without matching evidence in `outputs/`,
- silently dropping expected experiments,
- writing a polished report disconnected from real artifacts,
- ignoring the report template,
- ignoring `latex-writing-skill` when writing LaTeX,
- guessing which outputs belong to which experiment when the mapping is unclear.

## Required response pattern

When using this skill, structure the work and updates around this pattern:

### 1. Repository understanding
Summarize the expected experiments, output conventions, and report scaffold.

### 2. Expected experiment checklist
List the required experiments from README/project docs and the evidence each one should have in `outputs/`.

### 3. Output inventory
For each expected experiment, state:
- evidence found / partially found / missing / ambiguous
- artifact paths
- key metrics or files discovered
- any mapping uncertainty

### 4. Report updates
State which report sections can now be written or revised based on discovered artifacts.

### 5. Final report state
State whether `docs/report/report.tex` is now complete, partially complete, or blocked, and why.

## If artifacts are missing

If an expected experiment has no corresponding artifacts in `outputs/`:
- record that the experiment was expected,
- state what was looked for,
- do not fabricate substitute results,
- include the absence/limitation in the report,
- only suggest rerunning experiments as a follow-up if the user wants that.

## If the README is incomplete

If the README does not clearly enumerate all required experiments, infer them conservatively from:
- the project plan,
- implementation plan,
- report outline,
- repository configs/scripts,
- output naming conventions already present in `outputs/`.

In that case, explicitly separate:
- experiments explicitly documented in README,
- experiments inferred from other project sources.

## Final instruction

Use the repository documentation to determine what experiments the final report should cover, inspect the already existing files in `outputs/` as the sole evidence base for results, and write the final grounded report in `docs/report/report.tex` using the provided report structure and `latex-writing-skill` conventions.
